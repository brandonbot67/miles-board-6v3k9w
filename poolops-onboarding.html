1. SMS-Ready One-Pager (read this part only if you read nothing else)

**Recommendation: do NOT build a symmetric "choose your path" flow. Mail is the only activation track that can finish in the first session. Receptionist cannot — it needs an admin CLI run today. Ship mail-to-proof; make receptionist a started, not finished promise.**

The three facts that decide this:

  1. Mail is one step from activation. /get-started already does company → service area → Lob-verified return address (components/GetStartedWizard.tsx:9). It ends on a saved form, not a proof. The owner never sees a thing that looks like mail. The whole gap is one screen: show them their postcard.
  2. Receptionist is not self-serve at all. There is no provisioning API route. It runs through scripts/onboard-tenant.mjs — a white-glove CLI Brandon runs by hand, which prints a "Twilio TFV submission checklist" (scripts/onboard-tenant.mjs:1-33). No self-serve path exists to build on.
  3. The receptionist clock is regulatory, not engineering. Toll-free verification gates SMS only — not voice (Twilio docs). Twilio publishes no approval SLA. PoolOps already models this: sms_provider='manual-onetap' is set "legally safe until their TFV/A2P clears" (scripts/onboard-tenant.mjs:20). You cannot promise "receptionist live today" no matter what you build.

What this means: the two tracks are not peers. Track A ends in a dopamine hit in ~5 minutes. Track B ends in "we're on it, check back." A choose-your-path screen that offers both as equals sets up half your first users to hit a dead end on day one — the exact opposite of activation.

Recommended shape — one shell, one track, one deferred promise:

Signup → company → area → address (exists today) → NEW: "Here's your postcard" proof screen → cockpit with a 3-item checklist where "Turn on your AI receptionist" is item 3 and opens a request form, not a wizard.

What Brandon decides next — pick one:

  • (A) Ship-now, ~1 screen of work: add the postcard proof screen at the end of /get-started. Uses the existing Lob preview capability. This is the highest value-per-hour move in this doc.
  • (B) Ship-now+, ~1 screen + 1 form: A, plus a "Request my receptionist number" form that queues to Brandon's batch instead of pretending to provision. Honest, and it starts the TFV clock on day one.
  • (C) Build: make receptionist genuinely self-serve (provisioning API + number search UI + TFV form). Multi-day. Not recommended before customer #1.

My call: (B). It's roughly a day, it makes the mail track feel real, and it converts the receptionist from a broken promise into a scheduled one.

4. Mapping to PoolOps — the two tracks are not symmetric

Track A — Marketing mail live · ~90% built

What exists (verified):

  • Wizard: company → area → address, Step = 'company' | 'area' | 'address' | 'done' (components/GetStartedWizard.tsx:9,37)
  • Copy is already good and human: "Where should mail come from?", "Towns or zip areas you cover — so we know your market when you scan." (components/GetStartedWizard.tsx:223-229)
  • Lob return-address verification wired in (app/api/onboarding/first-run/route.ts, lib/lob-return-address-verify.ts)
  • Undeliverable addresses correctly block completion (lib/tenant-onboarding.ts:82-95)
  • Postcard send, templates, preview, scheduling, credits all real (app/api/pool-scans/[id]/postcards, /preview, app/api/mail-schedules, app/api/mail-credits)

⚠️ Model correction — this is not "address book → campaign → send." PoolOps mail is pool-scan-anchored: you scan an area/job, and postcards target prospects near it (app/api/pool-scans/[id]/postcards, /mail-blitz). Segmentation is segment_query against existing data. There is no standalone contact-import UI. Designing an address-book-first onboarding would mean building a product that doesn't exist. The task brief assumed "address book → campaign → first piece"; the codebase says otherwise, and the codebase wins.

The one real gap: the wizard's last screen is a form submission. The owner has never seen a postcard. Every reference product in §2 ends first-run on a rendered artifact.

Track B — AI receptionist live · not self-serve; regulatory clock

What exists (verified):

  • Twilio provisioning: lib/twilio-provision-core.js (raw REST, has dry-run SIDs) — called only by scripts/onboard-tenant.mjs and lib/twilio-provision.ts
  • Per-tenant "brain" (system prompt), GET/POST versioned (app/api/cockpit/receptionist/brain/route.ts:2-3) — schema notes a new tenant is "config, not a code change"
  • Voice/SMS/transcription/recording/voicemail all real (app/api/twilio/voice-inbound, app/api/calls/*)
  • Guardrail tests treat it as production-critical (package.json:59)

What does NOT exist (I checked — an earlier automated scan claimed these and was wrong):

  • ❌ No app/api/cockpit/receptionist/preview route
  • ❌ No app/api/cockpit/receptionist/autobuild route
  • ❌ No provisioning API route anywhere under app/api (find app/api -ipath "provision" -o -ipath "number" → empty)
  • ❌ No owner-facing receptionist page — only app/admin/receptionist/page.tsx (admin, not tenant)
  • ❌ Zero mentions of "receptionist" anywhere in app/get-started, app/signup, app/api/onboarding, or the wizard

Actual path today: owner signs up → Brandon runs scripts/onboard-tenant.mjs by hand → submits Twilio TFV → waits. This is white-glove by design ("Founding 5 era", scripts/onboard-tenant.mjs:3).

The compliance nuance that creates an opportunity: TFV gates SMS to US/Canada, and the docs do not state that voice is restricted (Twilio). So answering calls is plausibly not TFV-blocked — only texting back is, which is exactly why PoolOps ships sms_provider='manual-onetap' as the legally-safe interim. ⚠️ Verify with Twilio before promising this in copy — I could not find an explicit "voice is unrestricted" statement, and inferring permission from silence in a compliance doc is how people get numbers blocked.

Highest-leverage receptionist idea (borrowed from Fin, costs no phone number): let the owner chat with their receptionist brain in-app before any number exists. The brain is already GET/POST-able per tenant. That converts a multi-day regulatory wait into a first-session aha — they hear their receptionist before they own a number. This is the one receptionist feature worth building early, and it's a text box, not a telecom integration. (Flagged as gap vs. ship-now — it's real work, just unusually cheap work.)

5. Recommended First-Run Wireframe (plain English, screen by screen)

Shell: keep the existing /get-started rail and its step-chip header. No choose-your-path screen — one spine, receptionist enters as a checklist item after the win.

Screen 1 — Company (exists, keep as-is)

  • H1: "Welcome, {FirstName}." · Sub: "Confirm the company details for your workspace — about 30 seconds."
  • Fields: business name, phone. Server skips this when signup prefilled it (components/GetStartedWizard.tsx:87-90) — good, keep.
  • Success: name + phone saved.

Screen 2 — Service area (exists, keep as-is)

  • H1: "Where do you work?" · Sub: "Towns or zip areas you cover — so we know your market when you scan."
  • Success: serviceArea non-empty.

Screen 3 — Return address (exists, keep — but change the button)

  • H1: "Where should mail come from?"
  • Lob verifies deliverability; undeliverable = cannot complete (correct, keep).
  • Change: button text from a generic finish → "See my postcard →". Sets the expectation that a thing is coming.
  • Success: returnAddressComplete === true + completion stamp.

Screen 4 — NEW: "Here's your postcard." ← the activation moment

  • H1: "Here's your postcard." · Sub: "This is what lands in a mailbox near your next job. Nothing has been sent."
  • Body: rendered postcard preview — their business name, their return address, a default template. Reuse the existing preview path (app/api/pool-scans/[id]/postcards/preview, app/api/postcard-templates).
  • Explicit safety copy: "Nothing has been sent" — mirrors Lob/PostGrid test-mode norms. Owners fear accidentally mailing 500 postcards; say the quiet part.
  • Primary CTA: "Scan an area to send these →" (into the real pool-scan flow — the actual product spine)
  • Secondary: "I'll do this later" → cockpit. Never trap.
  • Success criterion: owner sees a rendered artifact bearing their own business name. This is the aha. Instrument it.
  • Gap flag: needs a preview callable with no pool scan yet (today's preview hangs off pool-scans/[id]). Either seed a demo scan for the tenant's service area or add a template-only preview. This is the one piece of build work in the ship-now path — scope it before committing.

Screen 5 — Cockpit with a 3-item checklist (pattern: 3–7 items, value-ordered)

  1. ✅ Your mail identity is ready — done, shown checked (progress-from-the-start; borrowed from standard checklist practice)
  2. ⬜ Send your first postcards — → pool scan flow
  3. ⬜ Turn on your AI receptionist — → Screen 6

Screen 6 — Receptionist: a request, not a wizard (honest track B)

  • H1: "Get a number that answers when you're in a pool."
  • Body — say the real timeline: "We set your number up for you and submit the carrier paperwork. Texting needs carrier approval, which takes a few days and isn't something we control. We'll tell you the moment it clears."
  • Do NOT show a fake number-search UI. There is no provisioning API; a UI that implies self-serve is a lie the backend can't cash.
  • Action: form (business name, desired area code, current number to forward) → queues to ~/shared-brain/batch-queue/, not to Twilio. Brandon runs scripts/onboard-tenant.mjs. This is white-glove, correctly labeled.
  • Success: request queued + honest ETA shown.
  • Stretch (the Fin move): "Hear your receptionist now" → chat with the tenant brain via the existing brain route. Aha with zero telecom.
